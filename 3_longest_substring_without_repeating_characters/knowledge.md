# Go言語における文字列(string)と文字(rune)の扱いについて

Go言語において、文字列をループで1文字ずつ処理する際に `rune` から `string` へのキャスト（変換）が必要になる理由とそのベストプラクティスについて解説します。

## なぜキャスト（string(charRune)）が必要なのか？

Go言語において、文字列（`string`）は「読み取り専用のバイト（`byte`）のスライス」として扱われます。
しかし、`for _, c := range s` のように `range` を使って文字列をループ処理すると、Goは文字列を自動的にUTF-8としてデコードし、1文字ずつ **`rune`（`int32`のエイリアス）** 型として取り出します。

### 型の不一致が発生する
文字列の先頭部分を保持している変数（例：`currentSubstring`）は `string` 型です。
一方、ループから取り出した1文字（例：`charRune`）は `rune`（数値）型です。

Go言語は型に非常に厳格な言語であり、**暗黙的な型変換を行いません**。
そのため、以下のような操作を行うとコンパイルエラーになります。

1. **文字列との結合**: `string` + `rune` はエラー（`string` + `string` はOK）
2. **stringsパッケージの関数の引数**: `strings.Contains()` や `strings.Index()` の第2引数は `string` 型を期待するため、`rune` を渡すとエラーになります。

したがって、`rune` として取り出された1文字を、文字列の一部として結合したり、`strings` パッケージの関数で検索したりするためには、**明示的に `string()` でキャストして `string` 型に戻す必要がある** のです。

## これはベストプラクティスなのか？

結論から言うと、**今回のケース（文字列の部分文字列を文字列として管理し、`strings` パッケージの機能を使いたい場合）においては、`string(charRune)` でキャストするのは一般的で正しいアプローチ（ベストプラクティスの一つ）** です。

他にもアプローチはありますが、目的によって使い分けます。

### アプローチ1: runeを取り出してstringに変換する（現在のコード）

```go
for _, charRune := range s {
    charStr := string(charRune) // ここで都度stringとして変換
    if strings.Contains(currentSubstring, charStr) { ... }
    currentSubstring += charStr // 文字列同士の結合
}
```
**メリット:**
- `strings` パッケージの豊富な関数（`Contains`, `Index`など）をそのまま利用できる。
- コードが直感的で読みやすい。

**デメリット:**
- 毎回 `string()` によるメモリアロケーション（新しい文字列の生成）が発生するため、文字列が非常に長く、パフォーマンスが極度にシビアな環境では微小なオーバーヘッドになる可能性がある。

### アプローチ2: スライスとして処理する（byteを扱う）

今回のLeetCodeの問題の制約が **「英語、数字、記号、スペースのみ（つまりすべてASCII文字であり1文字1バイト）」** であることが保証されている場合、`range` を使わずにインデックスで直接アクセスし、`byte` をそのまま扱う方法も高速です。少し高度になりますが、`strings.IndexByte()` などを組み合わせます。

```go
// s[i] は byte (uint8) を返す
for i := 1; i < len(s); i++ {
    charByte := s[i]
    // strings.IndexByte に byte をそのまま渡せる
    if idx := strings.IndexByte(currentSubstring, charByte); idx >= 0 {
        currentSubstring = currentSubstring[idx+1:]
    }
    // += による追加ではなく切り出しを使ったり、byteのまま保持するなど工夫が必要
    // （この例だと結局 currentSubstring += string(charByte) のように変換が必要になるか、文字列構築の工夫が必要）
}
```

### まとめ

一般的なアプリケーション開発や、マルチバイト文字（日本語など）が混ざる可能性のある文字列処理においては、`range` で安全に `rune` を取り出し、`string()` でキャストして `strings` パッケージの恩恵を受けるのが**安全で最も推奨されるベストプラクティス**です。

LeetCodeのように「文字がASCIIのみ」「ミリ秒・バイト単位の最適化が必要」といった特殊な状況下では、すべてをスライス（`[]byte`）ベースで計算する等のゴリゴリの最適化コードを書くこともありますが、まずは現在の「読みやすく、標準パッケージを活用した実装」がGo言語らしい書き方と言えます。
